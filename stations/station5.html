My Viewport Query Disaster

After Station 4, I felt unstoppable! I had intrinsic grids that adapted automatically. Time to build the perfect card component. I wanted it "responsive," so I used viewport media queries:

My "perfect" responsive card:

  display: flex;
  flex-direction: column;
  padding: 1rem;
  gap: 1rem;
}

/* Viewport query: "Is screen ‚â• 768px?" */
@media (min-width: 768px) {
  .card {
    flex-direction: row;

  }

/* Translation: "Desktop layout when viewport is wide" */

Then I built a page with a sidebar. Main content area (900px wide): Card looked great! Sidebar (300px wide): I placed the SAME card component there...

Text overflowed. Image squished. Padding too big for narrow space. The card was asking "How wide is the SCREEN?" (1440px) when it should have been asking "How wide is MY CONTAINER?" (300px)!

Same card, same viewport, different containers‚Äîbut viewport queries can't tell the difference!

Container Queries Fundamentals

Prof. Teeters sat me down to explain why viewport queries are extrinsic and container queries are intrinsic:

Why Viewport Queries Are Extrinsic

They ask a global question: "How wide is the SCREEN?"

@media (min-width: 768px) {
  .card { /* desktop layout */ }
}

/* Card in main (1200px viewport): Desktop ‚úì
   Card in sidebar (1200px viewport): Desktop ‚úó
   Viewport is same, containers different! */

Problem: Component can't know its ACTUAL space. It only knows screen size!

Why Container Queries Are Intrinsic

They ask a local question: "How wide is MY CONTAINER?"

@container (min-width: 400px) {
  .card { /* layout when container ‚â• 400px */ }
}

/* Card in main (700px container): Desktop ‚úì
   Card in sidebar (300px container): Mobile ‚úì
   Adapts to ACTUAL space given! */

Solution: Component is self-aware! Adapts to container, not viewport!

Container query setup pattern:
/* Step 1: Parent becomes a container */
.card-container {
  container-type: inline-size;  /* Query inline dimension (width) */
  container-name: card;         /* Optional: name for specificity */
}

/* Step 2: Children query their container */
@container card (min-width: 400px) {
  .card {
    flex-direction: row;  /* Horizontal when container ‚â• 400px */
  }
}

/* Works in ANY context: sidebar, main, full-width! */

Then Prof. Teeters showed me something HUGE: container UNITS!

Container Units (The Game-Changer!)

  • cqi - 1% of container inline-size (width in horizontal writing)
  • cqb - 1% of container block-size (height)
  • cqmin - Smaller of cqi or cqb
  • cqmax - Larger of cqi or cqb

Why this matters: Spacing can scale with ACTUAL container width, not viewport!

Why Container Units > Viewport Units

/* Station 3: Viewport units */
.card {
  padding: clamp(1rem, 0.5rem + 1vw, 1.5rem);
  /* Scales with viewport - same everywhere */
}

/* Station 5: Container units */
.card {
  padding: clamp(1rem, 0.5rem + 2cqi, 3rem);
  /* Scales with CONTAINER - adapts to space! */
}

Sidebar (300px): Compact padding. Main (900px): Generous padding. SAME CSS!

HAP's Container Units Mind Explosion:

Me: "Wait... my SPACING can respond to the container width?!"

Prof. Teeters: "Exactly! Remember Station 3's computed slope pattern? clamp(min, base + slope √ó vw, max)? Same formula with container units: clamp(1rem, 0.5rem + 2cqi, 3rem)!"

Me: "So padding scales with ACTUAL container, not screen? A card in a narrow sidebar gets compact spacing, same card in wide main content gets generous spacing‚Äîautomatically?!"

Prof. Teeters: "That's the magic! Container-relative spacing with computed slopes. It's intrinsic spacing taken to the next level!"

Container-relative spacing pattern (computed slopes!):
.card {
  /* Same pattern as Station 3, but with container units */
  padding: clamp(1rem, 0.5rem + 2cqi, 3rem);
  /* Formula: base (0.5rem) + slope (2% of container width) */

  gap: clamp(0.5rem, 0.25rem + 1.5cqmin, 2rem);
  /* Based on smaller container dimension */
}

/* Why computed slope instead of just cqi:
   - clamp(1rem, 5cqi, 3rem) - jumps unpredictably
   - clamp(1rem, 0.5rem + 2cqi, 3rem) - smooth scaling ‚úì */

Just like viewport units from Station 3, use base + slope √ó cqi pattern for predictable scaling!

Custom properties for container breakpoints:
:root {
  /* Container breakpoints (like viewport breakpoints) */
  --container-sm: 400px;
  --container-md: 600px;
  --container-lg: 900px;
}

.card-container {
  container-type: inline-size;
}

@container (min-width: var(--container-sm)) {
  .card {
    flex-direction: row;  /* Horizontal when container ‚â• 400px */
  }
}

@container (min-width: var(--container-md)) {
  .card {
    gap: var(--space-lg);  /* More spacing at medium+ */
  }
}

Systematic container breakpoints using custom properties‚Äîmaintainable and semantic!

Intrinsic Media

Container queries solved component layout. But Prof. Teeters asked: "HAP, what about images and videos? They need to adapt too!" She taught me intrinsic media properties:

aspect-ratio - Maintain Proportions

.video-wrapper {
  aspect-ratio: 16 / 9;
  /* Always 16:9 regardless of container size */
}

.square-image {
  aspect-ratio: 1;
  /* Always square (1:1) */
}

.portrait-image {
  aspect-ratio: 3 / 4;
  /* Portrait orientation */
}

Why intrinsic: No fixed dimensions! Ratio maintained automatically as container changes!

object-fit - How Content Fits

  • cover - Fill container, crop edges (intrinsic - container defines boundary)
  • contain - Fit inside, letterbox if needed
  • fill - Stretch to fill (extrinsic - distorts content!)
  • scale-down - Smaller of contain or none

Most common: object-fit: cover for images that fill space beautifully!

object-position - Cropping Control

img {
  object-fit: cover;
  object-position: top left;
  /* Crop from top-left */
}

img {
  object-position: center;
  /* Default: crop from center */
}

img {
  object-position: 50% 25%;
  /* Custom: horizontal center, 25% from top */
}

Smart cropping: Control which part of image shows when space is tight!

Responsive images without fixed dimensions:
img {
  max-width: 100%;           /* Never overflow container */
  height: auto;              /* Maintain aspect ratio */
  aspect-ratio: 16 / 9;      /* Explicit ratio */
  object-fit: cover;         /* Fill space, crop edges */
}

/* Result: Image adapts to ANY container!
   - Sidebar (300px): Image 300px wide, maintains 16:9
   - Main (900px): Image 900px wide, maintains 16:9
   - No media queries, no JavaScript! */

Fully intrinsic images‚Äîrespond to container, maintain proportions, never distort!

Custom properties for aspect ratios:
:root {
  /* Systematic aspect ratios */
  --aspect-wide: 16 / 9;
  --aspect-square: 1;
  --aspect-portrait: 3 / 4;
  --aspect-ultrawide: 21 / 9;
}

.hero-image {
  aspect-ratio: var(--aspect-wide);
  object-fit: cover;
}

.avatar {
  aspect-ratio: var(--aspect-square);
  object-fit: cover;
}

.product-photo {
  aspect-ratio: var(--aspect-portrait);
  object-fit: cover;
}

Semantic naming makes media systematic and maintainable!

Complete Intrinsic Component

Complete intrinsic component (combining all 5 stations):

  container-name: card;

  display: flex;

.card:has(img) {
  padding: clamp(1.5rem, 0.75rem + 3cqi, 4rem);
  /* More padding when has image, scales with container! */

/* Station 5: Container query for horizontal layout */
@container card (min-width: var(--container-sm)) {
  .card {
    flex-direction: row;
    gap: var(--space-md);

.card img {
  aspect-ratio: var(--aspect-wide);
  object-fit: cover;
  max-width: 100%;
  height: auto;
}

Why This Is Fully Intrinsic

  1. Container queries (St5): Responds to actual space given
  2. Container units (St5): Spacing scales with container width
  3. :has() selector (St3): Spacing adapts to content presence
  4. aspect-ratio (St5): Media maintains proportions
  5. Custom properties (St1): Systematic and maintainable

Self-Aware Component in Action

Same HTML, same CSS‚Äîperfect adaptation!
