Understanding AI drift

After 5 stations of learning modern CSS, I was excited to use AI to help me code faster. I imagined asking for layouts and getting perfect intrinsic designs instantly. Reality was... different. Why AI defaults to older CSS patterns

Training data is historical, not cutting-edge

AI tools learn from millions of code examples. Most of those examples are from 2010-2020 when fixed pixels, hex colors, and float layouts dominated. Container queries (2023), :has() selector (2023), and widespread intrinsic sizing patterns (2020+) are NEWER than most training data. Prof. Teeters explained: "The AI isn't wrong‚Äîit's working with what it learned. If 90% of layout examples in training used width: 300px, that's the AI's default. Modern techniques are less common in training data, so AI needs explicit guidance to use them."

AI reverts to most common patterns

When I gave vague prompts like "create a card grid," AI generated the MOST COMMON card grid pattern from its training‚Äîfixed columns with repeat(3, 1fr), fixed widths, viewport media queries. All extrinsic! Without context about OUR project's intrinsic layout standards, AI couldn't know I wanted repeat(auto-fit, minmax(16rem, 1fr)) with container queries. It defaulted to what it had seen most often. Prof. Teeters: "That's drift. Not AI making mistakes‚ÄîAI working with incomplete context. Our job is to provide the missing context through project rules and clear prompts."

# HAP Intrinsic Layout Project - CSS Standards

## CRITICAL: Read these rules FIRST before generating any code

### CSS Colors (MANDATORY)
- ALWAYS use hsl() format: `hsl(210, 50%, 50%)`
- NEVER use hex (#3498db) or rgb()
- NEVER use named colors (blue, red, etc.)
- Define colors as CSS custom properties in :root
- Example: `:root { --primary-color: hsl(210, 76%, 54%); }`

### Layout Philosophy (from Stations 1-5)
- **Primary**: CSS Grid for two-dimensional layouts
- **Secondary**: Flexbox for one-dimensional layouts
- **Intrinsic sizing**: Use `minmax()`, `auto-fit`, `auto-fill`
- **Container queries**: Prefer `@container` over `@media` for components
- **Fluid spacing**: Use `clamp()` with computed slopes (base + vw), NEVER arbitrary percentages

### Spacing Standards (Station 3)
- Use CSS custom properties for spacing: `--space-xs` through `--space-xl`
- ALWAYS use `clamp()` for fluid spacing with computed slope formula
- NEVER use fixed pixels: NO `padding: 24px`
- Example: `padding: clamp(1rem, 0.5rem + 1vw, 1.5rem)`

### Grid Patterns (Station 4)
- Use magic pattern: `repeat(auto-fit, minmax(16rem, 1fr))`
- NEVER use fixed column counts: NO `repeat(3, 1fr)`
- Use `fr` units, not percentages
- Example: `grid-template-columns: repeat(auto-fit, minmax(var(--grid-item-min), 1fr))`

### Container Queries (Station 5)
- Components use `@container`, NOT `@media`
- Set `container-type: inline-size` on parent
- Use container units (`cqi`, `cqmin`) for component-relative spacing
- Example: `padding: clamp(1rem, 0.5rem + 2cqi, 3rem)`

### Terminology (IMPORTANT)
- Say "CSS custom property" NEVER "CSS variable"
- Say "intrinsic layout" NEVER "responsive layout"
- Say "container query" NOT "element query"

### HTML Standards
- Use semantic HTML5 elements (<header>, <nav>, <main>, <footer>)
- Avoid unnecessary wrapper divs
- Maximum 3 levels of nesting for layout containers
- All images need alt, width, height attributes

### Accessibility (WCAG 2.2 Level AA REQUIRED)
- All interactive elements keyboard accessible
- Proper heading hierarchy (no skipping levels: h1 ‚Üí h2 ‚Üí h3)
- Color contrast: 4.5:1 for normal text, 3:1 for large text
- ARIA labels on navigation landmarks

## What to AVOID (Old CSS patterns)
- ‚ùå Fixed pixel widths: NO `width: 300px`
- ‚ùå Fixed pixel spacing: NO `padding: 24px`
- ‚ùå Hex colors: NO `#3498db`
- ‚ùå rgb/rgba: NO `rgb(52, 152, 219)`
- ‚ùå Named colors: NO `background: blue`
- ‚ùå float or clearfix patterns
- ‚ùå Viewport media queries in components: NO `@media (min-width: 768px)` for cards/widgets
- ‚ùå Fixed Grid columns: NO `repeat(3, 1fr)`
- ‚ùå Excessive nesting: NO 4+ wrapper divs

## Modern CSS Examples (USE THESE)

### Color System
```css
:root {
  --primary-color: hsl(210, 76%, 54%);
  --text-color: hsl(0, 0%, 20%);
  --bg-color: hsl(0, 0%, 98%);
}

.button {
  background: var(--primary-color);
  color: var(--text-color);
}
```

### Intrinsic Grid Pattern
```css
.grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(16rem, 1fr));
  gap: clamp(1rem, 0.5rem + 1.25vw, 2.5rem);
}
```

### Container Query Pattern
```css
.card-container {
  container-type: inline-size;
}

.card {
  padding: clamp(1rem, 0.5rem + 2cqi, 3rem);
}

@container (min-width: 400px) {
  .card {
    flex-direction: row;
  }
}
```

I keep this file in EVERY project. When AI reads these rules before generating code, it matches my modern CSS style automatically. No more drift to fixed pixels and hex colors!

Strategy 2: Plan first approach

The plan-first workflow

  1. Ask: "Give me a short implementation plan using the project rules file"

  3. YOU review plan for drift signals (fixed pixels? hex colors? media queries?)
  4. Approve plan or request corrections if drift detected
  5. Then request: "Now implement that plan" ‚Üí code follows approved plan

The self-review prompt

Why this works

  2. AI compares output to rules systematically (checks ALL rules, not just what you noticed)
  3. Often catches drift initially missed (AI generated fast, didn't verify on first pass)
  4. You see corrections - AI explains what it missed, you learn the patterns
  5. Creates teaching moment - Understanding AI's violations improves your own drift detection

HAP's rules for working with AI

After months of working with AI for intrinsic layouts, I've developed six non-negotiable rules. These aren't suggestions‚Äîthey're requirements that Prof. Teeters approves and enforces. Following these rules is the difference between AI being helpful versus AI being harmful. Ignore them, and you'll ship code you don't understand. Follow them, and AI becomes a powerful collaborative tool for writing modern CSS faster while maintaining professional standards.

1

Never ship code you don't understand

If AI generates code and I can't explain what every line does, I don't ship it. Period. Prof. Teeters taught me: "HAP, you own every line of code, whether you wrote it or AI did. If something breaks, users don't care who generated the bug‚Äîthey care that you shipped it." I read AI-generated code line by line. If I don't understand a pattern, I ask AI to explain it. If explanation doesn't make sense, I request a simpler approach or research the technique. Understanding beats speed. Always.

2

Always request accessibility explicitly

AI consistently forgets accessibility unless I explicitly request it. I learned this the hard way when Prof. Teeters found my "perfect" card grid had zero keyboard navigation and color contrast violations everywhere. Now my prompts ALWAYS include: "WCAG 2.2 Level AA compliant with keyboard navigation." I verify with Lighthouse audit (target: 100/100 accessibility). If AI's output isn't accessible, I request rewrite emphasizing accessibility first. Prof. Teeters' rule: "Accessibility is not optional. It's not a nice-to-have. It's a requirement. Every. Single. Time." I never forget now.

3

Project rules file prevents drift

I maintain .github/copilot-instructions.md (or CLAUDE.md for Claude Code) with project CSS standards: hsl() colors, container queries, clamp() spacing, intrinsic Grid patterns. This file is ALWAYS applied to AI requests automatically. I update it when I learn new patterns or catch drift repeatedly. Rules file is single source of truth for AI context. Without it, AI drifts to training data defaults (fixed pixels, hex colors, viewport queries). With it, AI generates modern CSS matching my project automatically. Write rules once, prevent drift forever. Non-negotiable.

4

Plan before code catches drift early

I NEVER ask AI for code directly. I always request plan first: "Give me implementation plan using project rules." I review plan for drift signals (hex colors mentioned? fixed widths? media queries?). If plan shows drift, I correct at planning stage (takes 10 seconds). If plan looks good, I approve and request: "Now implement that plan." Code follows approved plan, dramatically reducing drift. Prof. Teeters: "Catching drift at planning stage saves 20 minutes per feature." She's right. Plan-first approach is fastest way to get modern CSS from AI. Always plan, always review, then code.

5

Two-layer verification is mandatory

Before accepting ANY AI code, I run two-layer verification: (1) My 10-second drift scan‚Äîvisual check for fixed pixels, hex colors, viewport queries, excessive nesting (2) AI self-review‚Äî"Check your last response against project-rules file. List violations and fix them." My scan catches obvious drift. AI self-review catches subtle violations I missed. Together they catch 99% of drift before code enters my codebase. Takes 20 seconds total. Saves hours of debugging. Prof. Teeters: "Verification isn't overhead‚Äîit's how professionals work with AI." She's right. Generate ‚Üí Scan ‚Üí Self-review ‚Üí Accept. Every time.

6

Test before committing (always!)

AI-generated code that looks perfect in editor can break in browser. I ALWAYS test before committing: (1) Visual test‚Äîresize browser 320px to 1920px, verify responsive behavior (2) Keyboard test‚Äîtab through interactive elements, verify accessibility (3) Lighthouse audit‚Äîtarget: Performance 99+/100, Accessibility 100/100 (4) Console check‚Äîzero errors/warnings. If any test fails, I don't commit. I request AI fix or investigate manually. Prof. Teeters caught me committing untested AI code once. Never again. Her rule: "Test everything like you wrote it yourself. Because you're responsible for it either way." Testing is non-negotiable.

When NOT to use AI

AI is powerful for intrinsic layouts, but it's not always the right tool. Prof. Teeters taught me to recognize situations where manual work is better than AI assistance. Using AI when you shouldn't creates dependency on tools for things you should understand deeply yourself. Understanding when NOT to use AI is just as important as knowing when to use it. When debugging complex issues

The principle: Debugging builds understanding. Outsourcing it prevents learning.

I had a layout where Grid items overlapped unpredictably. Tempting to paste code into AI: "fix this." Prof. Teeters: "HAP, stop. This is a teaching moment. Debug it yourself using DevTools. Now I make architectural choices confidently and use AI to IMPLEMENT my decisions, not make them for me.

When to work manually: Choosing layout systems, designing component hierarchies, planning responsive strategies, accessibility approaches. Decide architecture yourself, use AI to implement decisions.
